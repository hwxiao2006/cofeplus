# 远程操作音量调节设计

## 背景

设备详情页当前的远程操作弹层里已经有 `音量调节` 入口，但点击后仍按普通远程指令直接执行，没有后续配置步骤。本次需要将其改造成带二级选择和三级调节页的交互流程，并贴近提供的移动端截图。

## 目标

- 在设备详情页的远程操作中，为 `音量调节` 增加后续操作步骤。
- 支持两个调节对象：`设备音量` 和 `点单屏音量`。
- 每个对象提供 `0-15` 的可调节范围，支持滑杆和 `- / +` 控件联动。
- 记住每台设备各自最近一次保存的音量值。
- 点击 `保存` 后停留在当前页，仅提示成功，同时记录远程操作日志。

## 非目标

- 不接入真实设备接口。
- 不改造除设备详情页远程操作之外的页面。
- 不新增白底识别、联网校验或异步执行状态回传。

## 交互流程

### 1. 入口层

用户在设备详情页打开 `远程操作` 后，仍看到现有一级菜单：

- 机构重启
- 设备开门
- 更新配方
- 设备停售
- 音量调节

点击 `音量调节` 后，不立即执行命令，而是进入二级菜单 `音量调节 · <设备编号>`。

### 2. 二级菜单

二级菜单仅展示两个入口：

- 设备音量
- 点单屏音量

点击任一入口后进入对应三级调节页。

### 3. 三级调节页

两个调节页共用同一布局结构，仅标题与说明文案不同：

- `设备音量`
  - 说明文案：`待机背景音乐、饮品出品信息、排队号语音`
- `点单屏音量`
  - 说明文案：`欢迎语音、下单提示语音、制作中音乐`

页面内容包含：

- 标题
- 对应说明文案
- 滑杆
- `- / 当前值 / +` 控件
- `保存` 按钮

## 状态设计

沿用现有远程操作弹层状态机，新增音量相关状态：

- `root`
  - 远程操作一级菜单
- `restart`
  - 机构重启二级菜单
- `confirm`
  - 机构重启确认页
- `volume-menu`
  - 音量调节二级菜单
- `volume-device`
  - 设备音量调节页
- `volume-order-screen`
  - 点单屏音量调节页

同时新增一个当前音量调节上下文对象，用于记录：

- 当前设备 ID
- 当前调节类型
- 当前编辑值

## 数据持久化

新增本地存储键保存音量设置，结构如下：

```json
{
  "RCK088": {
    "deviceVolume": 12,
    "orderScreenVolume": 6
  }
}
```

规则如下：

- 键名独立于 `devicesData`，避免污染现有设备主数据结构。
- 以设备 ID 为一级索引。
- `deviceVolume` 对应 `设备音量`。
- `orderScreenVolume` 对应 `点单屏音量`。
- 当某台设备无历史值时，默认使用 `0`。

## 行为规则

- 音量范围固定为 `0-15`。
- 点击 `-` 时最小不低于 `0`。
- 点击 `+` 时最大不高于 `15`。
- 拖动滑杆会同步更新当前数值显示。
- 点击 `保存` 时：
  - 写入本地存储
  - 追加一条远程操作日志
  - 显示成功 toast
  - 停留在当前调节页，不自动返回或关闭弹层

## 操作记录

保存成功后，写入现有远程操作记录流，动作名与备注建议为：

- 动作名：`设备音量调节`
  - 备注：`<设备编号> 设备音量已调整为 <值>`
- 动作名：`点单屏音量调节`
  - 备注：`<设备编号> 点单屏音量已调整为 <值>`

这样可以与已有设备远程操作记录保持一致，同时让日志可直接体现保存值。

## 视觉与实现约束

- 延续当前设备详情弹层体系，不新开整页视图。
- 三级调节页在视觉上尽量贴近提供截图的垂直结构和按钮层级。
- 图标可使用现有页面内联图标或轻量 SVG，不引入额外资源依赖。
- 保持桌面端与移动端共用同一逻辑；桌面端继续使用当前对话框容器，移动端继续使用底部弹层容器。

## 测试范围

至少新增或更新以下运行时回归测试：

- 点击 `音量调节` 后进入二级菜单，而不是直接执行命令。
- 二级菜单包含 `设备音量` 与 `点单屏音量` 两个入口。
- 点击两个入口后分别渲染对应说明文案和默认值。
- 默认值会读取该设备最近一次保存值。
- `-`、`+`、滑杆三种方式会同步更新当前值，并遵守 `0-15` 边界。
- 点击 `保存` 后：
  - 保持弹层停留在当前页
  - 写入本地存储
  - 追加远程操作记录
  - 显示成功 toast

## 风险与控制

- 风险：在现有 `handleDetailRemoteAction` 内继续堆分支会让逻辑变乱。
  - 控制：将音量相关渲染、读取、更新、保存拆成独立辅助函数，主分发函数只负责状态切换。
- 风险：本地存储写入失败导致 UI 与持久化不一致。
  - 控制：沿用现有页面对 `localStorage` 的容错方式，失败时至少保持当前 UI 可用并提示保存失败。
- 风险：保存后停留当前页，用户可能连续重复保存。
  - 控制：toast 文案明确显示最新保存值，避免误判是否生效。
