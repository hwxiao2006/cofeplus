故障行展开方案对比

当前问题:故障行虽然可以点击展开,但只有左侧一个小三角(▸)作为提示,客户感知不到。 下面 3 个方案都用同一份 mock 数据(RCK386 / 咖啡机未检测到 SD 卡), 让你看哪个交互最让客户「自然知道这一行有更多信息」。

A · 推荐

第一步常驻显示 + 「查看全部」链接

第一条处理步骤始终可见,后面的步骤折叠。客户一眼就知道「这行有处理建议,我可以展开看全部」。

如何处理 · 完整步骤
  1. 当压力 ≥ 2.3 bar 时触发
  2. 检查压力传感器是否故障
  3. 联系维修排查泄压阀

优势

  • 感知最强:第一条步骤永远可见,客户立即明白「这有处理建议」
  • 对运维有帮助:常见故障一眼解决,不用点开
  • 「查看全部 N 步」的 N 数字也是 hint,告诉用户后面还有内容

代价

  • 每行变高 ~18px,屏幕能显示的行数变少
  • 故障内容列字数变多,要注意截断
B

「如何处理」按钮 in 操作列

在「操作」列加一个白底描边按钮 ▾「如何处理」。最像 web 应用的标准做法,按钮显式可点。

  • 仍然要点一次才能看到处理建议,运维不能直接执行常见操作
C

右侧「处理提示」常驻色块

在表格右边加一个新列(琥珀色卡片),专门放第一句处理提示。整张卡片可点击展开看完整步骤。

优势

  • 视觉权重最高:琥珀色卡片放在右边,客户一定看得到
  • 「💡 提示」语义最自然
  • 明确的步骤数 (2步 / 3步) 让用户判断要不要点开

代价

  • 表格列数从 5 增加到 6,1280px 以下屏幕会挤
  • 琥珀色和现有 teal 主色不完全协调
  • 移动端必须重新设计(纵向堆叠)
我的推荐:方案 A。理由:故障处理本质上是「一句提示就能解决 60% 问题」的场景, 把第一步永远露出来,运维(或客户自己)看一眼就知道做啥,不用每次都点。 方案 B 标准但效率低,方案 C 视觉冲击大但和现有 teal 风格冲突。
